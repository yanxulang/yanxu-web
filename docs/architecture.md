# yanxu-web 架构

框架只负责应用编排，不重复实现 HTML 转义或 HTTP 编解码。依赖方向始终单向：

```text
应用代码
   │
   ▼
yanxu-web ──> yanxu-html
   │
   └────────> yanxu-http
```

`yanxu-html`和`yanxu-http`彼此独立；`yanxu-web`同时依赖二者；示例应用依赖全部三层。这一结构允许协议测试不启动框架，也允许 HTML 组件用于邮件或静态生成。

## 请求生命周期

1. `Web开发服务器`从监听器接受一个 TCP 连接；
2. `yanxu-http`按预算读取并解析一个`HTTP请求`；
3. `Web应用.处理`用方法与路径查询路由器；
4. 框架创建`Web上下文`，填入路径参数、共享状态和允许方法；
5. 中间件链按注册顺序进入，最终调用路由、404 或 405 端点；
6. 端点返回`HTTP响应`，中间件可以在返回途中补首部；
7. 任意处理错误进入统一错误处理器；
8. `yanxu-http`编码响应，服务器发送后关闭连接。

## 中间件模型

每次请求创建独立的中间件链对象，游标不会跨请求共享。中间件签名为`法（上下文，下一步）`：

```text
中间件 A 进入
  中间件 B 进入
    路由处理器
  中间件 B 返回
中间件 A 返回
```

中间件可以实现响应首部、简单认证、计时或请求标识。0.1 没有异步并发、请求取消与流式正文，因此不应把长任务放进开发服务器请求路径。

## 错误边界

路由解析、中间件和端点执行共享一个捕获边界。默认错误处理器返回 500，不泄露错误内容。自定义错误处理器获得当前上下文和捕获的`误`，但也必须返回`HTTP响应`。

服务器层无法解析请求时会尝试发送 400；若客户端已经断开，则进入统一关闭路径。业务错误应在应用错误处理器中映射，协议错误由服务器层处理。

## 静态文件边界

静态路由只允许末尾通配参数映射到文件根下的普通文件。路径规范化前先拒绝危险段和字符，避免目录穿越；未知扩展使用`application/octet-stream`。框架不会生成目录列表，也不跟随 URL 中的隐式索引页约定。

## 扩展方向

稳定接口优先保持应用、上下文、处理器和响应工厂的概念不变。并发服务器、长连接、流式正文、Multipart、WebSocket、HTTP/2 与 HTTP/3 将随`yanxu-http`底层能力分阶段进入；生产部署能力还需要日志、指标、优雅关闭、可信代理和 TLS 策略。当前范围见[文档站路线图](https://docs.yanxu.dev/web/security-roadmap/)。
